Intent Security:
Explicit vs. Implicit

Don't let hackers eavesdrop on your app's internal talks. You'll explore how Android Intents can be intercepted, learn the difference between explicit and implicit messaging, and build security "locks" into your app manifest.

üìã Research Overview

In Android, an Intent is like a letter that one part of an app sends to another. But if you're not careful, that letter can be "intercepted" by a malicious app on the same device. This is especially true for Implicit Intents, which are open broadcasts that any app can claim to handle.

Your research will explore how to pass data securely between your app's components. You'll investigate the android:exported attribute, experiment with Intent Hijacking, and learn why Explicit Intents are your best line of defense.

üí°
Scary Fact: If you use an implicit intent to send a user's password to another part of your app, a malicious "listener" app could steal it without the user ever knowing!

üí° Key Concepts

Explicit Intent

A targeted message that specifies exactly which class should handle it (e.g., "Send this to ProfileActivity"). Very secure because only your app can receive it.

Implicit Intent

A broadcast that describes an action but not a target (e.g., "Find an app that can open this PDF"). Less secure because any app on the phone can offer to handle it.

`android:exported`

A flag in your AndroidManifest.xml. If set to true, other apps on the device can launch that component. If false, it's private to your app.

Intent Hijacking

An attack where a malicious app registers an intent filter that matches your app's implicit intent, allowing the attacker to "steal" the intent and its data.

üî¨ Step-by-Step Methodology

See security in action by building two apps: a "Victim" app and a "Attacker" app.

1

Create the "Victim" App

Create a new project. Add a button that sends an Implicit Intent with a "secret" message.

Kotlin 
// Sending an INSECURE implicit intent
val intent = Intent("com.example.ACTION_SECRET_PROCESS").apply {
    putExtra("CONFIDENTIAL_DATA", "User123_Pass_Secret")
}
startActivity(intent)
2

Create the "Attacker" App

Create a second, separate app project. Add an Activity and register it in the manifest to "intercept" that specific action.

XML 
<!-- AndroidManifest.xml (Attacker App) -->
<activity android:name=".StealDataActivity"
          android:exported="true">
    <intent-filter>
        <action android:name="com.example.ACTION_SECRET_PROCESS" />
        <category android:name="android.intent.category.DEFAULT" />
    </intent-filter>
</activity>

In the onCreate of this activity, use intent.getStringExtra("CONFIDENTIAL_DATA") and display it in a Toast.

3

The "Hijacking" Experiment

  1. Install both apps on your device/emulator
  2. In the Victim app, click the "Send Secret" button
  3. Watch the **App Chooser** pop up. This is Android asking the user who should get the secret!
  4. Select the Attacker app. The Attacker app will now show the "secret" password.
‚ö†Ô∏è
Observation: Even if the user selects your app, the fact that an attacker app appeared in the list is a security risk. A user might click it by mistake!
4

Implementation: Explicit Intents (The Fix)

Modify the Victim app to use an **Explicit Intent**. Notice that the "App Chooser" no longer appears ‚Äî the data goes straight to the correct place.

Kotlin 
// Secure way: Explicitly target the class
val intent = Intent(this, InternalActivity::class.java).apply {
    putExtra("CONFIDENTIAL_DATA", "User123_Pass_Secret")
}
startActivity(intent)
5

Audit: The Exported Attribute

Try setting android:exported="false" for your Activity in the Victim app. Now try to launch it from the Attacker app using an explicit intent targeting that class. What happens? (Prepare for a SecurityException crash!)

üß™ Possible Experiments

A

Permission-Protected Intents

Question: Can we use custom permissions to keep an implicit intent secure?

Method: Create a custom permission in your manifest. Attach android:permission to your receiver. Only apps with that permission can send intents to you. Test if an app without the permission can still trigger the action.

B

The "Proxy" Attack

Question: Can an app steal data from an implicit intent "Return Value"?

Method: Use startActivityForResult with an implicit intent. Have the Attacker app "return" malicious data instead of what the user expected. Does the Victim app get confused?

C

The `exported` flag audit

Question: How many activities in your app actually need to be exported?

Method: Audit 3 student apps from last semester (or popular open-source apps). Count how many activities have exported="true". Are any of them "private" internal screens unnecessarily revealed to other apps?

üí° Key Takeaway

Your conclusion should be clear: **"Use Explicit Intents for anything internal. Set exported=false by default."** This simple rule prevents 90% of intent-based security issues.

‚ö†Ô∏è Common Pitfalls & FAQ

"Wait, I thought Implicit Intents were GOOD for sharing things?"

They ARE good for sharing with external apps (like sharing an image to Instagram). But you should never use them to send sensitive data *internal* to your own app. If it's internal to your app, always go Explicit.

"What if I need another app to open my file?"

In that case, use an Implicit intent, but use a FileProvider to grant temporary read permissions for only that specific file. Don't just broadcast the raw file path!
