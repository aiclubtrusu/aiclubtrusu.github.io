üìã Research Overview

Images are usually the largest part of any Android app's APK file. However, many developers simply drag-and-drop raw high-resolution PNGs or JPEGs into their project without realizing the cost. This leads to massive download sizes and high memory usage, which can make apps feel slow or unstable on older devices.

In this research, you will conduct an "Asset Audit." You'll compare different image formats, resolutions, and scaling strategies to find the "Sweet Spot" where image quality remains high but file size and memory footprint are minimized. You'll learn to use professional tools like **APK Analyzer** and **Android Profiler** to prove your findings with hard data.

üí° Key Concepts

WebP Format

A modern image format from Google that provides superior **lossless and lossy** compression for images on the web and Android. It can be 30% smaller than PNGs with the same quality.

Screen Density (DPI)

Android devices have different pixel densities. You need to provide assets for `mdpi`, `hdpi`, `xhdpi`, etc., to ensure images look sharp without wasting pixels on lower-end screens.

VectorDrawable (SVG)

Instead of a grid of pixels, these use XML to define shapes. They scale to any size perfectly and take up almost zero space. Perfect for icons and logos!

Bitmap Decoding

The process of turning a compressed file (like a JPEG) into a raw array of color values in RAM. This is where memory leaks and crashes usually happen.

üî¨ Step-by-Step Methodology

Conduct a comparative analysis of asset efficiency.

1

The "Raw" Initial Build

Find 5 high-resolution photographs (4K PNGs). Create an Android project and add them all to your `res/drawable` folder. Build the APK and record its file size.

2

Analyze with APK Analyzer

Go to **Build ‚Üí Analyze APK...** in Android Studio and select your APK. Drill down into the `res` folder to see exactly which images are taking up the most space. Take a screenshot of the "Top Offenders."

3

WebP Conversion Sprint

Right-click your images in the `res/drawable` folder and select **Convert to WebP...**. Choose "Lossy" at 75% quality. Re-build the APK and compare the new size to the baseline in Step 1.

4

Memory Profiling

Run your app on a device/emulator and open the **Android Profiler (View ‚Üí Tool Windows ‚Üí Profiler)**. Click on the **Memory** graph. Load your large images and watch the "Graphics" memory spike. Try this with unoptimized vs. optimized assets.

5

Vector vs. Raster Icon Test

Create a 512x512 PNG icon and a VectorDrawable version of the same icon. Zoom in 500% on both in your layout editor. Record the file size difference and the quality difference when scaled.

üß™ Possible Experiments

A

The "Resolution Overkill" Test

Question: Does using a 4K image in a 100x100 `ImageView` waste memory?

Method: Display a huge image in a small container. Measure memory in the Profiler. Then, resize the image to exactly 100x100 and repeat. **Expect a massive difference!**

B

PNG vs. WebP Efficiency Map

Question: At what quality percentage does WebP start looking "Bad"?

Method: Convert the same image to WebP at 100%, 80%, 50%, and 20% quality. Track the file size at each step. Create a visual scale for your poster showing the "Visual Transparency Point."

C

The Adaptive Density Audit

Question: How much space is wasted by including `xxxhdpi` assets for all devices?

Method: Bundle high-res assets in the `drawable-nodpi` folder vs density-specific folders. Research the benefit of **Android App Bundles (.aab)** in solving this problem automatically.

üîÑ Alternative Approaches

Network Dynamic Loading

Instead of bundling images inside the APK, research loading them from a URL using **Glide** or **Picasso**. Does this technically solve the APK size problem, or just move the problem to the user's data plan?

R8 and ProGuard Resource Shrinking

Android Studio can automatically remove resources that you aren't actually using in code. Research enabling `shrinkResources true` in your `build.gradle` and measure the savings on a messy project.

‚ö†Ô∏è Common Pitfalls & FAQ

"Why does my app crash when I scroll through a list of images?"

This is the classic **OutOfMemoryError**. You are likely decoding too many large Bitmaps at once. Research "Bitmap Pooling" and how library like Glide handles this for you automatically.

"Can't I just use JPEGs for everything?"

JPEGs do not support **transparency (alpha channels)**. If your icon needs a transparent background, you must use PNG, WebP, or VectorDrawable. Also, JPEGs often have more "noise" around sharp text than WebP.

"Is WebP supported on all Android versions?"

Basic WebP is supported since Android 4.0, but Lossless/Transparent WebP requires Android 4.3+. Since 99%+ of devices are on much newer versions now, it is safe to use everywhere!
