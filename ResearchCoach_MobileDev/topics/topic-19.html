üìã Research Overview

When you write code in Kotlin or Java, the Android phone doesn't "read" your text files directly. It first translates them into machine code that the processor understands. This process is called **Compilation**. Historically, Android developers have fought a war between two methods: **JIT (Just-In-Time)** and **AOT (Ahead-Of-Time)**.

JIT is fast to install but can make the app "stutter" during the first run as it compiles code on the fly. AOT makes the app run perfectly from the start but makes installation painfully slow. Modern Android uses a sophisticated hybrid called **Profile-Guided Optimization (PGO)**. In this research, you will explore this evolution and measure the "Cold Start" impact of compilation patterns.

üí° Key Concepts

JIT (Just-In-Time)

The compiler translates code **while the app is running**. It's fast to install but consumes extra CPU and battery during execution, which can lead to "Jank."

AOT (Ahead-Of-Time)

The compiler translates the entire app **during installation**. This ensures maximum speed during use but results in massive installation times and larger binary sizes.

ART (Android Runtime)

The modern execution environment for Android. It uses a hybrid approach: JIT results are "profiled" and then compiled via AOT in the background while the phone is charging.

Baseline Profiles

A list of the most important code paths provided by the developer. This tells ART which parts to AOT-compile immediately during installation to prevent that first-run lag.

üî¨ Step-by-Step Methodology

Investigate the "Self-Optimizing" behavior of your Android device.

1

Select a "Complex" App

Choose an app from the Play Store that you haven't used recently (or clear its cache/data). This ensures you are starting from a "Cold" compilation state.

2

Measure "Cold Start" Latency

Force-stop the app. Open it and use a stopwatch (or your eyes) to measure how long it takes to reach the main interactive screen. Record any stuttering or frame drops during the first 30 seconds of use.

3

The "Training" Phase

Open and close the app 10 times. Navigate through all the menus. This allows the JIT profiler to learn which parts of the code are "Hot" (used frequently).

4

Simulate Background AOT

Normally, Android compiles the "Hot" code overnight. You can force this using **ADB (Android Debug Bridge)** command to speed up your research:

adb shell cmd package compile -m speed-profile [package_name]
5

Measure "Warm Start" Efficiency

Launch the app again. Is the startup faster? Does the scrolling feel smoother? Compare the "Cold" vs "Warm" metrics. This is your primary data for the poster.

‚ö†Ô∏è Common Pitfalls & FAQ

"Does AOT always make things faster?"

Usually, yes. However, AOT code takes up more **disk space**. If your device is almost full, the OS might delete AOT files to save room, making your app slow again. This is called "Profile Purgatory!"

"Can I disable JIT for my app?"

You can't easily disable it for end-users, but you can force-compile your app during development to see how it performs under perfect conditions.

"What is a .vdex file?"

It stands for "Verified DEX." It contains verified code that the JIT doesn't have to check again, speeding up execution. It's like a "Pre-checked" pass at an airport security line.
