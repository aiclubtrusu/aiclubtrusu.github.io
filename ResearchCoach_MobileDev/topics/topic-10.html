The Navigation Dilemma

üìã Research Overview

In the early days of Android, every single screen in an app was a new **Activity**. But launching a new Activity is "heavy" on the system ‚Äî it takes time and consumes more RAM. Today, most apps use a **Single Activity Architecture**, where one main activity stays alive and swaps out smaller pieces of UI called **Fragments**.

In this research, you'll build two versions of the same app: one using "Multi-Activity" and one using "Single-Activity + Fragments." You'll use the **Memory Profiler** to see which one uses more RAM and analyze how much easier it is to share data (like a user's login) when you only have one activity.

üí° Key Concepts

Back Stack

The "History" of screens you've visited. When you hit the "Back" button, the system "Pops" the current screen off the stack to show the previous one.

Fragment Transaction

The process of swapping one Fragment for another. Unlike starting an Activity, this happens inside the same "Context" and is much lighter.

Navigation Graph

An XML file that maps out all your app's screens and the "Actions" (arrows) that connect them. It makes complex flow easy to visualize.

Context Sharing

The ability for multiple fragments to "see" the same data. By sharing a `ViewModel` with the activity, you can pass info without using complex `Intent` extras.

üß™ Possible Experiments

A

The "Context Overhead" Test

Question: How much RAM does a "Blank" Activity really use?

Method: Build an app that launches 10 identical empty Activities. Record total RAM. Build another that launches 10 identical empty Fragments. The difference is the "Activity Overhead."

B

Transition Jank Measurement

Question: Are Fragment transitions smoother than Activity transitions?

Method: Use the **"Display GPU Rendering"** tool in Developer Options. Watch the "Blue Bars" as you switch screens. If the bar crosses the "Green Line," it means you dropped a frame. Which architecture drops fewer frames?

C

Boilerplate Count: XML vs. Kotlin

Question: Which architecture requires more "Manifest" work?

Method: Every Activity *must* be declared in the `AndroidManifest.xml`. Fragments do not. Count the lines in your Manifest for both versions. This is a "Maintainability" finding for your poster.

‚ö†Ô∏è Common Pitfalls & FAQ

"Wait, I thought Fragments were dead? People say they are complex."

Fragments had a bad reputation for a few years because their lifecycle is complicated. But the **Jetpack Navigation Component** fixed almost all of those issues by handling the "Transactions" for you. They are very much the industry standard now!

"What happens if my one Activity crashes?"

Yes, if the single activity crashes, the whole app closes. In a multi-activity app, sometimes the OS can restart you at a specific activity. However, a modern app should have stable navigation logic, making this a secondary concern compared to the performance benefits.
